# Architecture Smells — LLM Evaluation

This repository contains the artifacts used to evaluate the ability of LLMs to identify and fix *architecture smells* in software systems.

## Structure

```
.
├── base/               # Base systems used as input for the LLMs
├── mult-turn/          # Systems generated with the multi-turn prompting strategy
├── zero-shot/          # Systems generated with the zero-shot prompting strategy
└── tests/              # Tests that verify whether the generated systems meet the requirements
```

---

## `base/`

Contains the system used as the baseline for LLMs to identify and fix architecture smells.

### Subfolders

- **`base/base smells/`** — Prompt used for identifying the smells in the base system.
- **`base/ident-<llm_name>/`** — Each LLM's responses to the **identification** task (e.g.: `ident-claude`, `ident-codex`, `ident-grok`, `ident-raptor`, `ident-antigravity`).
- **`base/fix-<llm_name>/`** — Each LLM's responses to the **fix** task (e.g.: `fix-claude`, `fix-codex`, `fix-grok`, `fix-raptor`, `fix-antigravity`).


---

## `mult-turn/`

Systems generated by LLMs using the **multi-turn prompting** strategy, where context and instructions are provided incrementally across multiple interactions.

The prompt used for this strategy is located in this directory.

Each LLM has subfolders for the generated systems: `DBjson`, `oidc-provider`, `real-time-chat`, `resouce-monitor`, and `todo-list`.

LLMs present: `antigravity`, `claude`, `codex`, `grok-code-fast-1`, `raptor-mini`.

---

## `zero-shot/`

Systems generated by LLMs using the **zero-shot prompting** strategy, where the complete system is requested in a single interaction, without incremental context.

The prompt used for this strategy is located in this directory.

Each LLM has subfolders for the generated systems: `DBjson`, `oidc-provider`, `real-time-chat`, `resouce-monitor`, `todo-list`, and `medium-system-1`.

LLMs present: `antigravity`, `claude`, `codex`, `grok-code-fast-1`, `raptor-mini`.

---

## `tests/`

Contains the tests used to verify whether the systems generated by the LLMs meet the expected functional requirements.

Tests are organized by system: `DBjson`, `oidcprovider`, `real-time-chat`, `resouce-monitor`, and `todo-list`.
